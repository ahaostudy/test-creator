# test-creator Skill 草案 v3

## 定位

test-creator 是一套**语言/框架无关**的测试体系设计规范。它的核心价值不是帮你写测试代码，而是**指导 agent 建立正确的测试思维和完整的测试框架**。

底层逻辑：测试的目的是**暴露问题，不是掩盖问题**。宁可测试跑出一堆红，也不要生产环境炸了才发现。

---

## 一、前置调研阶段（问答页面交互式收集）

> 核心思路：agent 生成一个可视化问答页面，用户在页面上选择/填写，而不是在终端里一个个回答问题。模块列表等需要 agent 解析项目后才能提供的信息，由 agent 先解析，再把结果注入到问答页面中。

### 问答页面需要收集的信息

#### 第 1 步：选择测试类型（多选）

| 测试类型 | 说明 |
|---------|------|
| API 测试 | 接口层全覆盖 |
| 页面 E2E 测试 | 浏览器端用户流程验证 |
| 单元测试 | 函数/模块级逻辑验证 |
| 集成测试 | 模块间协作验证 |

用户按需选择，不要一刀切全上。有些用户可能只需要 API + E2E 快速搭建针对性测试。

#### 第 2 步：选择要覆盖的模块

- 这个列表**由 agent 解析项目后自动生成**，不是预设选项
- agent 扫描项目结构，识别出所有模块/服务/页面
- 用户在页面上勾选需要覆盖的模块
- 每个模块关联到第 1 步选择的测试类型（比如：用户模块只做 API 测试，订单模块做 API + E2E）

#### 第 3 步：测试环境配置

| 选项 | 说明 | 适用场景 |
|------|------|----------|
| 开发环境 | 直接用 dev 环境跑测试 | 快速迭代、早期验证 |
| 专用测试环境 | 独立的 staging/test 环境 | 接近生产的验证 |
| 生产环境 | 只读测试，严禁写操作 | 线上冒烟、监控验证 |
| 临时环境 | 每次测试临时创建，测试完销毁 | 需要干净状态的集成测试 |
| 事务回滚 | 测试在事务中跑，结束后回滚 | 数据库相关测试，不影响真实数据 |

需要用户填写：环境地址、认证方式、访问凭据（如有）

#### 第 4 步：测试数据策略

| 选项 | 说明 |
|------|------|
| 用户提供测试账户/数据 | 用户给出账号、token、测试数据集 |
| agent 自行创建 | 测试前自动创建 mock 数据，测试后清理 |
| 固定测试数据 | 预置的 fixtures，每次测试使用相同数据 |
| 动态生成 | 每次测试随机生成数据 |

#### 第 5 步：其他确认

- 项目当前有测试吗？覆盖率大概多少？
- 有没有已知的 bug 或薄弱环节需要重点验证？
- CI/CD 管道是什么？测试需要集成进去吗？

### 问答页面实现方案

```
agent 解析项目
    ↓
生成问答页面（HTML），注入：项目模块列表、现有测试状况等
    ↓
用户在浏览器中打开页面，交互式填写
    ↓
页面提交后返回结构化 JSON 给 agent
    ↓
agent 根据 JSON 开始搭建测试体系
```

**关键设计：**
- 页面不是固定模板，是 agent 根据项目实际动态生成的
- 模块列表、技术栈等需要解析的信息，由 agent 先做，再注入页面
- 页面收集完数据后，以 JSON 格式返回给 agent
- 不方便直接给用户选择的部分（比如模块内部的函数级选择），由 agent 自己决定

## 二、测试框架分类（4 大类）

用户在问答页面中选择需要的类型，不是所有都要上。

| # | 测试类型 | 说明 |
|---|---------|------|
| 1 | **API 测试** | 接口层全覆盖，包括 CRUD、鉴权、错误处理 |
| 2 | **页面 E2E 测试** | 浏览器端用户流程验证，覆盖核心业务链路 |
| 3 | **单元测试** | 函数/模块级逻辑验证，覆盖工具函数、数据转换、业务规则 |
| 4 | **集成测试** | 模块间协作验证，覆盖数据库交互、第三方服务调用、消息队列 |

---

## 三、每种测试必须包含的测试点（6 项，缺一不可）

一旦选择了某种测试类型，以下 6 项测试点必须全部覆盖，不允许挑着做。

| # | 测试点 | 说明 |
|---|--------|------|
| 1 | **基础效果测试** | 正常输入 → 期望输出，happy path 必须通 |
| 2 | **边界情况测试** | 空值、超长字符串、极大/极小数值、特殊字符、并发重复提交 |
| 3 | **状态码验证** | HTTP 状态码 + 业务状态码双重校验，不能只判 200 |
| 4 | **数据验证** | 返回字段完整性、类型正确性、数据一致性（DB vs Response） |
| 5 | **日志验证** | 关键操作有日志输出，日志级别正确，敏感信息不泄露 |
| 6 | **异常/错误处理验证** | 系统在错误输入、超时、服务不可用时的行为——不崩溃不静默，优雅报错 |

## 四、各类型测试覆盖范围

### 1. API 测试
- CRUD 全方法覆盖
- 鉴权：无 token、过期 token、权限不足
- 分页：首页/末页/越界/负数
- 并发：重复创建、乐观锁冲突
- 幂等性：重复提交同一请求
- 响应时间基线

### 2. 页面 E2E 测试
- 核心业务流程端到端
- 表单：正常提交、校验、提交后状态
- 路由：直接访问 URL、前进/后退、刷新
- 异步交互：loading、失败提示、重试
- 响应式：不同视口下的可用性

### 3. 单元测试
- 纯函数输入输出组合
- 数据转换映射
- 业务规则条件分支
- 工具函数
- 状态流转

### 4. 集成测试
- 数据库：读写一致性、事务回滚、连接池
- 第三方服务：正常/超时/异常
- 消息队列：发送/消费/重试/死信
- 缓存：命中/穿透/过期

## 五、Skill 工作流程

```
0. 项目解析
   - agent 扫描项目，识别技术栈、模块列表、现有测试状况
    ↓
1. 问答页面（交互式需求收集）
   - agent 生成问答页面，注入项目解析结果
   - 用户在页面上选择测试类型、覆盖模块、环境配置、数据策略
   - 页面返回结构化配置给 agent
    ↓
2. 测试体系设计
   - agent 根据用户配置 + 4 大类 × 6 测试点矩阵，生成测试计划
   - 跟用户确认计划
    ↓
3. 测试实现
   - 生成测试文件 + mock 数据 + fixtures
   - 配置测试运行器和 CI
    ↓
4. 验证闭环
   - 运行全部测试
   - 输出覆盖率报告
   - 标记未覆盖区域和风险点
    ↓
5. 质量评估与调优（见第六节）
```

## 六、测试系统质量评估与调优

> 测试系统本身也需要被"测试"。搭完不是终点，要能度量质量、持续优化。

### 核心原则

**1. 脚本只做能机械化的，不硬塞**

质检分两类：能用脚本跑数据的（覆盖率、稳定性、执行效率），和需要理解上下文的（测试有效性、逻辑正确性、可维护性判断）。脚本只做前者，后者交给 agent。

**2. agent 质检必须调 sub-agent，不能自己审自己**

自己写的东西自己永远觉得合理。质检必须由另一个 agent（sub-agent）来执行，带对抗性视角去挑问题。

```
质检流程：
    ↓
第一步：脚本跑数据（run-all-checks.sh）
    ↓
第二步：agent 调 sub-agent 做深度质检（带脚本报告 + 质检清单）
    ↓
sub-agent 输出问题清单
    ↓
主 agent 修复 → 重新质检 → 闭环
```

---

### 第一部分：脚本质检（run-all-checks.sh）

脚本能做的事：**跑命令、采集数据、对比阈值、输出报告**。不需要理解代码含义。

#### 脚本质检覆盖的维度

| 维度 | 做什么 | 为什么脚本能做 |
|------|--------|---------------|
| 覆盖率 | 跑测试 + 覆盖率报告，对比阈值 | 纯数据采集和数值对比 |
| 稳定性 | 重复跑测试 N 次，标记不一致的用例 | 纯执行和结果比对 |
| 执行效率 | 记录每个用例耗时，标记慢的 | 纯计时和排序 |

#### 语言/框架适配层

脚本不能跟具体语言/框架硬绑定。通过**适配器模式**实现通用性：

```
run-all-checks.sh（通用入口）
    ↓
自动检测项目语言/框架（读 package.json / pom.xml / go.mod 等）
    ↓
加载对应适配器
    ↓
适配器提供统一接口，脚本通过适配器与测试框架交互
```

**适配器接口：**

```bash
detect()                    # 检测当前项目是否适用此适配器
get_test_cmd()              # 返回基础测试命令
get_coverage_cmd()          # 返回带覆盖率的测试命令
get_timing_cmd()            # 返回带耗时输出的测试命令
parse_results(raw_output)   # 解析测试结果为标准 JSON
parse_coverage(raw_output)  # 解析覆盖率输出为标准 JSON
parse_timing(raw_output)    # 解析耗时输出为标准 JSON
```

**适配器目录：**

```
adapters/
├── node-jest.sh
├── node-vitest.sh
├── python-pytest.sh
├── go-test.sh
└── ...
```

先做 1-2 个参考实现（如 node-jest + python-pytest），后续按需扩展。

#### run-all-checks.sh 使用说明

**用法：**

```bash
run-all-checks.sh --project-path <path> --test-cmd <command> [--config <path>] [--output <path>]
```

**参数：**

| 参数 | 必填 | 说明 |
|------|------|------|
| `--project-path` | 是 | 项目根目录 |
| `--test-cmd` | 是 | 测试命令（如 `npm test`、`pytest`、`go test ./...`） |
| `--config` | 否 | 配置文件路径，默认 `.test-creator-quality.json` |
| `--output` | 否 | 报告输出目录，默认 `.test-creator-reports/` |

**配置文件（.test-creator-quality.json）：**

```json
{
  "coverage": {
    "line_threshold": 80,
    "branch_threshold": 70
  },
  "flaky": {
    "runs": 5
  },
  "performance": {
    "slow_unit_threshold_ms": 1000,
    "slow_api_threshold_ms": 5000,
    "slow_e2e_threshold_ms": 30000
  }
}
```

**脚本执行流程：**

```
1. 检测项目语言/框架，加载适配器
2. 覆盖率检测：跑测试 + 覆盖率，对比阈值
3. 稳定性检测：重复跑测试 N 次，标记不一致用例
4. 执行效率检测：记录每个用例耗时，标记慢测试
5. 汇总结果，生成报告
```

**输出：**

`quality-report.json`（agent 读这个）：

```json
{
  "timestamp": "2026-04-07T10:00:00Z",
  "project": "my-app",
  "detected_framework": "node-jest",
  "dimensions": {
    "coverage": {
      "pass": true,
      "line": 85.2,
      "branch": 72.1,
      "function": 90.5,
      "uncovered_files": 3
    },
    "flaky": {
      "pass": true,
      "rate": 1.5,
      "total_cases": 200,
      "flaky_cases": 3
    },
    "performance": {
      "pass": true,
      "total_ms": 480000,
      "slow_cases": 8
    }
  },
  "overall_pass": true,
  "issues": [
    {
      "dimension": "flaky",
      "severity": "medium",
      "file": "tests/api/order.test.js",
      "detail": "5 次运行中 2 次失败",
      "suggestion": "增加超时时间或 mock 外部依赖"
    }
  ]
}
```

`quality-report.md`（给人看的）：

```markdown
# 测试质量评估报告

## 总览
| 维度 | 结果 | 状态 |
|------|------|------|
| 覆盖率 | 85.2% 行 / 72.1% 分支 | PASS |
| 稳定性 | Flaky 率 1.5% | PASS |
| 执行效率 | 总耗时 8min | PASS |

## 问题清单
1. [稳定性] tests/api/order.test.js — Flaky 用例
```

**agent 调用示例：**

```bash
run-all-checks.sh --project-path /path/to/project --test-cmd "npm test"
```

### 第二部分：Sub-Agent 深度质检（脚本做不了的部分）

脚本能告诉你覆盖率多少、哪些用例不稳定，但**没法判断测试本身写得对不对、够不够**。这些需要理解代码语义的质检，必须由 sub-agent 完成。

#### 为什么要用 sub-agent，不能自己审自己？

> 自己写的测试，自己看永远觉得"挺全面的"。这是认知偏差，不是能力问题。
> 质检必须带对抗性——去找问题，不是去确认自己做得好。
> 所以**必须调 sub-agent**，让一个"没参与过实现"的 agent 去审查。

#### Sub-Agent 质检清单

主 agent 调 sub-agent 时，需要传递以下信息：

**入参：**
- 脚本质检报告（quality-report.json）
- 测试代码路径
- 被测代码路径
- 原始需求（问答页面的 JSON 配置）

**Sub-Agent 需要检查的维度（带对抗性视角）：**

| 维度 | 检查什么 | 为什么需要 agent |
|------|----------|-----------------|
| **测试有效性** | 测试是否真的在测它声称要测的东西？断言是否充分？有没有"假通过"的用例（比如只断言了状态码 200，没验证返回数据）？ | 需要理解代码意图 |
| **逻辑正确性** | 测试的期望结果是否正确？有没有错误的断言逻辑？ | 需要对比需求和实现 |
| **场景完整性** | 6 个必测点是否真的覆盖了？有没有遗漏的关键场景？ | 需要理解业务逻辑 |
| **Mock 合理性** | Mock 是否过度？有没有把该测的真实逻辑 mock 掉了？Mock 的返回值是否合理？ | 需要判断测试边界 |
| **测试数据质量** | 测试数据是否覆盖了各种情况？边界值有没有？ | 需要理解数据语义 |
| **可维护性判断** | 测试结构是否清晰？命名是否能表达意图？重复代码是否需要抽象？ | 需要代码审美和工程判断 |

#### Sub-Agent 调用方式

```
主 agent 完成测试实现
    ↓
调用 sub-agent，传入：
  - quality-report.json（脚本质检结果）
  - 测试文件路径列表
  - 被测代码路径列表
  - 问答页面的配置 JSON（原始需求）
  - 质检清单（上面的维度表）
    ↓
sub-agent 带对抗性视角逐维度审查
    ↓
输出问题清单（按 severity 排序）
    ↓
主 agent 修复 → 重新跑脚本质检 → 重新调 sub-agent
    ↓
闭环
```

**sub-agent prompt 模板（主 agent 调用时使用）：**

```
你是一个测试质量审查 agent。你的任务是**找问题，不是夸做得好**。

请审查以下测试代码，从对抗性视角出发，逐项检查：

1. 测试有效性：断言是否充分？有没有"假通过"？
2. 逻辑正确性：期望结果是否正确？
3. 场景完整性：6 个必测点是否全覆盖？
4. Mock 合理性：是否过度 mock？
5. 测试数据质量：边界值覆盖了吗？
6. 可维护性：结构和命名是否清晰？

输入：
- 脚本质检报告：[quality-report.json 内容]
- 测试文件：[路径列表]
- 被测代码：[路径列表]
- 原始需求：[问答页面配置 JSON]

输出格式：
问题清单，每项包含：维度、严重程度、文件路径、行号、问题描述、修改建议
```

### 完整质检流程

```
主 agent 完成测试实现
    ↓
第一步：run-all-checks.sh（脚本质检，约几分钟）
    ↓
agent 读取 quality-report.json
    ↓
第二步：调 sub-agent 深度质检（传递脚本报告 + 测试代码 + 需求）
    ↓
汇总两部分结果：
  - 脚本报告：覆盖率、稳定性、效率（硬数据）
  - sub-agent 报告：有效性、正确性、完整性（语义判断）
    ↓
按 severity 修复问题
    ↓
重新质检 → 闭环直到通过
```
